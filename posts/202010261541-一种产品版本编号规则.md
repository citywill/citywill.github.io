# 产品版本编号的标准规范

产品版本号，遵循以下版本规范：

1. 产品立项时要确认产品的主要版本号。该版本号和产品计划发布年月保持一致，一般为`yymm`形式（例如`2010`）。
2. 产品立项时要有产品计划，由产品负责人根据对产品特性和进度的预期分为多个次要版本（里程碑）。次要版本一般按照自然序号顺延。例如`2010.1`，表示的是包括一些重要特性的第一个里程碑发布版本。
3. 次要版本又可划分入多个阶段，包括 base、alpha、beta、rc、release 等。例如`2010.1`是一个内部测试版本，就可以命名为`2010.1_alpha`。
   - base：如果产品在立项前有工作基础，或者没有实质特性的基础研发工作，则标注 base 版本。通常 base 版本的次要版本号为`0`。
   - alpha：代表已经完成了部分重要特性的内部版本。该版本只允许部署到公司内部环境，禁止出厂。
   - beta：代表主要特性已经完成，正在增加次要特性或者修复缺陷。该版本可以部署到客户环境进行演示和试用，但禁止正式出厂。
   - rc：代表已经完成计划内特性，并且已经完成内部测试，不会存在致命缺陷，等待交付验证或结项。
   - release：完成结项，正式发布的版本。
4. 一个产品由一个或以上的程序组成，每个程序存放到单独的 git 仓库中。每个次要版本的发布，都伴随着相应的 release 分支，例如`release/v2010.1_alpha`。在完成次要版本的开发，将 release 分支合并到 master 时，必须创建 tag。tag 名称中包含最后一次 build 的最大 issue 编号。例如`v2010.1.123_alpha`。
5. 产品对外宣传用的版本号为 `[产品名] v[主要版本号]`，例如 `慧擎知识管理平台 v2010`。必要时，在内部沟通等场合，可采用最大的程序 tag 名作为完整产品的版本号，例如 `慧擎知识管理平台 v2010.1.123 release`。
