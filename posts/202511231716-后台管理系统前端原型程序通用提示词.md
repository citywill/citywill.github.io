# 后台管理系统前端原型程序通用提示词

##  术语

开发者在撰写提示词时，往往不试用标准术语。本文建议采用以下术语表述开发任务。

- 系统：程序最大的单位就是系统。一个系统至少包含若干功能。
- 子系统：子系统之间通常相互独立，每个子系统都可以做为一个独立的程序存在。通常这些子系统会通过统一鉴权允许用户使用同一账户访问。
- 栏目：栏目是一系列相关模块的组合。一个系统可以划分为多个、多级栏目。通常栏目本身没有具体功能，知识作为一个逻辑容器，将一系列模块划分到一起。
- 模块：模块可以看作是一系列相关功能的集合。例如对一类信息的増、删、改、查就是一个模块的四类常用功能。一些模块因为具备强关系，也可以组合成一个上级模块。
- 功能：系统要实现具体业务目标的最终载体。打开一个页面，展开一个弹窗，点击一个按钮，只要这些行为能够满足客户的预期，都可以叫做功能。

## AI理解

AI辅助开发的最小单元也是功能，但通常模块的信息会作为若干功能的通用上下文。依次类推。在AI辅助开发过程中，出于大模型上下文窗口限制，一般以模块为单元进行开发。

项目上下文一般包括如下信息：

- 项目结构：系统、子系统、栏目、模块的层级信息。只定义层级即可。
- 技术架构：包括技术路线、程序结构、开发风格等要求。

提示词一般包括如下信息：

- 模块介绍
- 模块相关的实体、关系及属性
- 模块的功能信息。前端程序需要描述控件，后端程序需要描述入参和出参

## 以前端程序为例

这里举一个前端界面的例子。前段界面开发是大模型比较擅长的部分，提示词也比较简练。

对于前端程序而言，一个模块要描述功能清单，比如列表、详情、新增、编辑、删除等功能。这些功能还涉及到公共的模块信息，例如实体属性、控件描述等。

以数据清洗模块为例，该模块主要包括数据清洗规则管理和数据清洗任务管理两个单元模块，分别对应的其实是数据清洗规则和数据清洗任务两个实体的增删改查操作的功能集合。

该模块的提示词如下：

```markdown
# 数据清洗模块功能说明

数据清洗、转换、映射的规则配置及实现结果，支持将原始模型的数据清洗到目标模型。

包括两个模块：

## 数据清洗规则管理

管理数据清洗规则。包括数据清洗规则列表、详情、新增、删除、编辑等功能

- 字段属性：
  - ID
  - 规则名称
  - 转换类型：缺失值处理规则、异常值处理规则、数据一致性规则、冗余数据处理规则、数据格式标准化规则
  - 规则描述
  - 启用状态
  - 清洗函数：从函数列表中选择一个函数
- 检索条件：
  - 名称（模糊搜索）
  - 类型（下拉菜单）
- 记录操作：详情、编辑、删除、启用/禁用（对应启用状态）

## 数据清洗任务管理

管理清洗任务，包括清洗任务列表、详情、新增、删除、编辑等功能。

- 字段属性：
  - ID
  - 任务名称
  - 原数据模型（从数据模型中选择关联）
  - 目标数据模型（从数据模型中选择关联，不能和原始数据模型一致）
  - 字段映射：可以增加原数据模型的字段和清洗规则
- 检索条件：
  - 名称：模糊所搜
  - 原始数据模型：单选
  - 目标数据模型：单选
- 记录操作：详情、编辑、删除、运行/停止（运行一段时间后自动结束，并弹窗提示任务完成）

```
