# 软件源代码管理规范

研发部门所有代码都必须存放在公司指定的代码仓库中。

## 1. 权限和角色说明

| 权限/角色 | 管理员 | 产品负责人 | 项目负责人 | 研发参与人 | 产品团队  | 实施团队 |
| --------- | ------ | ---------- | ---------- | ---------- | --------- | -------- |
| 默认角色  | owner  | reporter   | reporter   | reporter   | guest     | guest    |
| 产品群组  |        | maitainer  | developer  |            | reporter  |          |
| 产品仓库  |        | owner      | developer  |            |           |          |
| 项目群组  |        |            | owner      | developer  |           |          |
| 文档中心  |        | developer  | developer  | developer  | developer |          |

角色说明：

| 权限                         | owner | maitainer | develop | reporter | guest |
| ---------------------------- | ----- | --------- | ------- | -------- | ----- |
| 浏览群组、仓库目录           | √     | √         | √       | √        | √     |
| 浏览仓库 wiki                | √     | √         | √       | √        | √     |
| 浏览、下载仓库代码           | √     | √         | √       | √        |       |
| 提交代码                     | √     | √         | √       |          |       |
| 合并到 main 分支             | √     | √         |         |          |       |
| 修改仓库、群组信息           | √     | √         |         |          |       |
| 创建和管理子群组和仓库       | √     |           |         |          |       |
| 高级操作（归档、删除、移动） | √     |           |         |          |       |

## 2. 代码仓库项目分类存放要求

| 分组     | 说明             | 标题规则                   | 标题示例                                   |
| -------- | ---------------- | -------------------------- | ------------------------------------------ |
| 产品     | 存放产品代码     | 产品立项命名               | 智慧未成年人检察监督平台                   |
| 项目     | 存放项目代码     | 立项日期-项目名称-仓库用途 | 202209-安庆大观区-任职限制监督法律监督模型 |
| 解决方案 | 存放解决方案代码 | 解决命名                   | 大数据法律监督解决方案                     |
| 文档中心 | 存放团队文档     |                            |                                            |

## 3. 产品代码仓库的定义要求

| 事项         | 要求                                  | 示例                                |
| ------------ | ------------------------------------- | ----------------------------------- |
| 命名（路径） | 采用英文单词或词组，多个单词用"-"连接 | minor-protection-supervise-platform |
| 标题         | 采用产品全称命名                      | 未成年人保护监督平台                |
| 描述         | 说明该仓库中代码的用户、功能和价值    | 未成年人保护监督平台后端服务代码    |
| 分支管理     | 遵循 gitflow                          |                                     |
| README       | 描述完整兼具简洁                      |                                     |

产品代码仓库命名、标题的后缀细则：

1. 平台（platform）：平台具备以下特征：规模较大，包含多个子系统，并实现对多个子系统的集成；用户较复杂，包含多个角色，角色之间存在信息交换关系。
2. 系统（system）：系统一般可包含后端服务和前端界面程序，其中后端程序提供 Api 接口，用户通过前端界面实现对 Api 的执行。
3. 服务（service）：服务一般包含特定功能范围的若干 Api 接口以及 Api 调用文档，提供给其他程序在运行后依赖。
4. 模块（module）：模块一般包含特定功能范围的若干类库，提供给其他程序在程序运行前依赖。
5. 脚本（script）：脚本一般包含若干代码，可提供给运维人员人工执行，或通过任务调度系统自动执行。

## 4. 对 README 文档的要求

| 事项         | 要                                               | 不要                 |
| ------------ | ------------------------------------------------ | -------------------- |
| 前提         | 精心编写                                         | 默认或者没有         |
| 标题         | 产品或项目的完整名称                             | 没有或者简称         |
| 一句话描述   | 采用“用户、功能、价值”的一句话描述               | 没有                 |
| 仓库结构说明 | 和项目文件结构相对应的模块名称及简要说明         | 没有对应不上项目结构 |
| 快速入门     | 五行以内的使用说明                               | 没有或者内容太多     |
| 详细文档导航 | 至少包含安装部署手册、实施运维手册、用户使用手册 | 没有或全写到 README 里   |
